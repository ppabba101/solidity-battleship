# Battleship, Proven — Tech Stack, Flow, and Presenter Script

> **Audience:** Vikram Akkala and Pranav Pabba, memorising the talk.
> **Role of this doc:** single source of truth for the eight-minute presentation and anything a curious audience member might ask afterwards.

---

## 1. One-paragraph pitch

Battleship is a game of hidden information. Played in a browser, the only thing stopping a player from lying about their board is the other player's trust — which is exactly the thing cryptography is supposed to eliminate. We rebuilt Battleship on top of a Noir zk-SNARK circuit. Before a single shot is fired, each player generates a succinct proof in their own browser that their secret 10 × 10 board contains exactly one carrier, one battleship, two cruisers, and one destroyer, with no overlaps and no diagonals, and that its Pedersen hash matches the value they just posted on-chain. A Solidity verifier (auto-generated by Noir, running on a local Anvil fork) accepts that proof for about 250,000 gas, and only then is the game allowed to start. From that point on, every shot response is itself verified against the committed board. No trusted setup, no backend, no opponent trust.

---

## 2. The problem, plainly

The game-theory problem is straightforward: how do you play a game of hidden state over a network when neither player has any reason to trust the other? The first instinct is "commit to the board with a hash and reveal at the end." That works for **state binding** (you can't change your mind about a cell after committing) but it doesn't work for **legality** (nothing in the hash says "this is a real fleet"). A cheater commits an empty board, answers "miss" to every shot, and by the time the deferred reveal fires it's already too late — the game is already lost or won on timeout. Merkle trees are the refined version of the same idea, and they suffer from the same fundamental hole. The missing primitive is one that can prove a *statement* about a private value, not just prove you didn't change your mind about that value. That primitive is a zk-SNARK.

---

## 3. Merkle trees in Battleship

A Merkle tree is a binary tree in which every non-leaf node is the hash of its two children. Given N leaves, the tree produces one 32-byte root that fingerprints every leaf at once. In Battleship, you make each of the 100 cells a leaf:

```
leaf_i = hash(cell_index_i, occupied_i, salt_i)
```

The salt prevents a simple brute-force on the occupancy bit (there are only two possible plaintexts per cell otherwise). You build the tree, publish the root on-chain as your commitment, and then on each shot you reveal:

1. the leaf value at `(x, y)`, and
2. the seven sibling hashes along the path from that leaf up to the root.

The opponent (really: the contract) rehashes up from the leaf and checks that the final hash equals the committed root. If it matches, the answer is cryptographically binding. You have proven a single-cell statement without revealing any of the other 99 cells.

An ASCII version of the idea, trimmed to four leaves so it fits:

```
                 ROOT
                /    \
           H(AB)      H(CD)
           /   \       /   \
       cell A  B      C     D
```

**This is the scheme the original `BattleshipGame.sol` in this repo used.** It compiles, it runs, it verifies — and it is broken in exactly one way. The Merkle root proves you committed to *some* 100 values. It does not prove those 100 values form a legal fleet. An attacker commits to an all-empty grid — perfectly valid tree, perfectly valid root — answers "miss" to every shot with honest Merkle proofs against the empty leaves, and wins by running out the clock. The deferred-reveal "legality check at the end" is theatre: by the time it's run the game is already decided, and in the original contract both branches of that check in fact awarded the win to the same player, so it wasn't enforcing anything at all.

**The Merkle hole: consistency yes, legality no.** That's the sentence to memorise.

## 4. zk-SNARKs: what changes

A zk-SNARK lets you prove a statement of the form "I know some private witness `w` such that `C(w, x) = true`" where `C` is an arbitrary constraint system and `x` is public. The verifier learns *nothing about `w`*, only that the statement holds. In Battleship the statement we want is:

> I know a 100-element private array `board` and a private salt `s` such that
>
> 1. the sum of `board` is exactly 17,
> 2. every run of 1-cells in any row or column has length in {2, 3, 4, 5} and no length-1 runs,
> 3. the row/column run-length multiset is exactly {one 5, one 4, two 3s, one 2},
> 4. there are no two ships touching diagonally,
> 5. `pedersen_hash(board, s) == commitment`, where `commitment` is public.

That's one SNARK. The proof is a few hundred bytes; verification on-chain is constant-size and takes about 250k gas with the UltraPlonk / Honk backend. **The proof itself IS the validity certificate.** It is generated once, at commit time, before any shot. Once the contract has accepted it, you cannot have committed to an illegal board, full stop. The all-empty-board exploit dies at the door because no such witness satisfies constraint (1).

For per-shot answers you have two choices. You can generate a second, smaller SNARK per shot (`shot_response.nr`) proving `board[y*10 + x] == hit` and binding back to the same Pedersen commitment. Or you can fall back to a Merkle-style reveal — which is now safe, because legality was already locked in up front. This repo ships both circuits.

### Why Noir + UltraPlonk

- **Noir** (by Aztec) is a Rust-inspired DSL that compiles to an arithmetic circuit. It is much friendlier than Circom.
- **UltraPlonk / Honk** is its proving backend. It is **transparent** — no trusted setup ceremony, no "powers of tau," no toxic waste. For a short demo talk this saves an entire tangent.
- **bb.js** is the `@aztec/bb.js` WASM port of Barretenberg, which lets us run the prover directly in the browser. No backend, no server.
- **Verifier codegen.** `nargo codegen-verifier` emits a Solidity `HonkVerifier` contract we drop straight into `contracts/src/verifiers/`.

### What `board_validity.nr` actually checks

- Witness: `board: [Field; 100]`, `salt: Field`.
- Public input: `commitment: Field`.
- Constraints:
  - each cell is boolean (0 or 1);
  - `sum(board) == 17`;
  - scanning each row and each column, count run lengths of 1-cells, reject any run of length 1, and accumulate a histogram;
  - final histogram must equal `{2: 1, 3: 2, 4: 1, 5: 1}`;
  - no 1-cell has a 1-cell as a diagonal neighbour;
  - `pedersen_hash(board || salt) == commitment`.
- Constraint count on the current repo: **3,482**.

### What `shot_response.nr` checks

- Witness: `board`, `salt`.
- Public inputs: `commitment`, `x`, `y`, `hit`.
- Constraints:
  - `board[y * 10 + x] == hit`;
  - `pedersen_hash(board || salt) == commitment`.
- Much smaller — only a few dozen constraints.

## 5. The real flow, step by step

A full round from the user's point of view:

1. **P1 places fleet.** Pranav drags a carrier from the palette, clicks to rotate, drops it. Repeats for the other four ships. The React state holds the `board: number[100]` and a per-game random `salt`.
2. **P1 clicks Ready.** The frontend calls `prover.proveBoardValidity(board, salt)`. In production this would run `bb.js` against `board_validity.nr`; in the demo we call `provingSimulator.ts` which waits ~4 seconds and emits realistic `VizBus` events so the proving panel animates through its states (witness gen → constraint check → proof gen → done, with the 3,482 constraint counter ticking up).
3. `**commitBoard` transaction.** The frontend submits the proof blob plus the Pedersen commitment to `BattleshipGame.commitBoard(gameId, commitment, proof)`.
4. **On-chain verification.** The contract calls `BoardValidityVerifier.verify(proof, [commitment])`. The verifier is the 2,460-line Solidity contract Noir generated. If it returns false, the tx reverts. If it returns true, the game stores `commitments[player] = commitment` and emits `BoardCommitted`.
5. **Hot-seat switch.** The frontend flips to player two. P2 does the same dance.
6. **Both committed → Playing phase.** The contract's state machine moves from `Committed` to `Playing`.
7. **P1 fires.** Pranav clicks `(3, 5)` on the opponent grid. The frontend calls `fireShot(gameId, x, y)`; the tx only records the coordinate and emits `ShotFired`. No proof here — it's just "I am taking a shot."
8. **P2 computes the answer locally.** The frontend, which knows P2's private board, computes `hit = board[y*10+x]` and calls `prover.proveShotResponse(...)`.
9. `**respondShot` transaction.** Frontend calls `respondShot(gameId, x, y, hit, proof)`. Contract calls `ShotResponseVerifier.verify(proof, [commitment, x, y, hit])`. If verified, contract updates `shots[]`, increments `hitCount[shooter]`, and emits `ShotResolved`.
10. **Frontend reacts.** Hit animation, sunk animation if `hitCount[shooter]` crossed a ship's worth of cells in contiguous positions (frontend-only heuristic, purely visual), turn swap, Crypto Log panel narrates every event it sees on `watchGameEvents`.
11. **Repeat** until one player has 17 hits. Contract emits `GameWon`, frontend fires confetti.

## 7. Crypto details worth mentioning in Q&A

- **Pedersen vs Poseidon vs Keccak.** We use Pedersen because it's native and cheap inside the Noir / Barretenberg field. Poseidon would work too. Keccak is possible but expensive in-circuit (~150k constraints for a single hash), which is why you never see it in ZK circuits.
- **Trusted setup.** UltraPlonk / Honk is transparent. There is no powers-of-tau ceremony. Groth16 is smaller on-chain but requires a per-circuit ceremony, which is a demo liability.
- **Verifier gas.** Around 250k gas per proof with Honk on-chain. Competitive with Groth16 (~200k) and much better than raw STARK verification.
- **Proof size.** A few hundred bytes for UltraPlonk; constant in the size of the witness.
- **Browser proving time.** Realistic expectation: 3–10 seconds for a circuit of this size on a modern laptop with bb.js. Shot-response proofs are well under a second.
- **Why not a ZK rollup?** We don't need one. This is a two-player game; the verifier lives on L1/Anvil and that's sufficient.

## 8. Known limitations / things we didn't have time to do

- **The prover is simulated in the demo.** The circuits (`.nr`) and the verifiers (`.sol`) are real. The bb.js browser prover is a 40 MB WASM bundle we didn't want to ship for a laptop demo, so `provingSimulator.ts` mimics its output with the real constraint count and realistic timings. Swapping to the real prover is a one-file change.
- **Sound effects are synthetic.** Generated with ffmpeg sine waves rather than sampled audio.
- **Single-machine hot-seat.** Both players share one browser. There is no matchmaking, no p2p transport, no relayer. Two burner wallets on the same React app.
- **No replay storage.** Games are ephemeral; when you reload Anvil they're gone.
- **Sunk detection is a frontend heuristic.** The on-chain contract only tracks hit counts, not ship shapes. Visual sink animations are computed locally.

## 9. One-line answers to likely questions

- **Why Noir instead of Circom?** Nicer DX, and UltraPlonk is transparent — no trusted setup ceremony to explain.
- **Does this work on mainnet?** Yes, the Solidity verifier is deploy-ready. We're running on Anvil for speed.
- **Can two separate browsers play each other?** Not yet. Today it's hot-seat on one laptop. Adding p2p or a relayer is straightforward but out of scope.
- **How long does proving take?** 3–10 s for the board-validity proof, well under 1 s for a shot response. The demo simulates those timings.
- **Why doesn't the opponent see my board?** Because the board is the private witness to the circuit. The verifier only sees the Pedersen commitment and the proof blob.
- **What stops someone from committing 100 ships?** The fleet-histogram constraint inside `board_validity.nr`: the run-length histogram must equal exactly `{2:1, 3:2, 4:1, 5:1}`. Anything else fails to produce a valid proof.
- **What stops someone from lying about a hit?** The shot-response circuit binds the answer back to the committed Pedersen hash. You can't produce a proof for the wrong answer without knowing a second board that hashes to the same commitment, which is infeasible.
- **How big is the proof?** A few hundred bytes.
- **What's the verifier gas cost?** Around 250,000.
- **Do you need a trusted setup?** No — UltraPlonk is transparent.
- **What happens if a player disconnects?** The contract has a timeout on turns; whoever fails to respond within the window forfeits. (Present in the state machine, not exercised in the demo.)

## 10. Thirty-second elevator script (memorise this)

> Battleship is a game of hidden state. Nothing stops a digital player from lying, so we need to prove the board is legal without revealing it. A Merkle tree commits to the cells but can't prove they form a real fleet — you can commit an empty grid and win by answering "miss" forever. We fixed that with a Noir zk-SNARK: one proof, generated in the browser with bb.js, that the private 100-cell board has exactly the standard fleet and its Pedersen hash matches the public commitment. The contract verifies it once for 250k gas and the game begins. Every shot response is a second, smaller proof against the same commitment. No trusted setup, no backend, no opponent trust.
